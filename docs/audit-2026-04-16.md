# Technical Audit of pmsimstats-ng

## Scope

This document reports a comprehensive technical audit of the
pmsimstats-ng repository, covering the installed R package at the
repository root, all four implementations under `implementations/`,
the strategic documentation under `docs/`, and the driver scripts
under `analysis/`. The audit examines structural inventory,
statistical methodology, code quality, evolution across
implementations, and publication readiness.

Repository path: `/Users/zenn/Dropbox/prj/alz/10-pmsimstats-ng/pmsimstats-ng/`

Package name: `pmsimstats` (version 0.3.0)

Four implementations inspected:

| Collection | Style | Architecture | Location |
|---|---|---|---|
| original | data.table | B (MVN) only | `implementations/original/R/` |
| original-extended | data.table | A + B | `implementations/original-extended/R/` |
| tidyverse | tidyverse | A + B | `implementations/tidyverse/R/` |
| simple | base R plus tidyverse | A only | `implementations/simple/simulation.R` |

## 2. Statistical Model Comparison

### 2.1 Response decomposition

All four implementations decompose the outcome into three latent
factors: true biological response at time on drug (BR), placebo or
expectancy response at cumulative expectancy time (PB), and
time-varying nuisance response at cumulative trial time (TV). The
three are summed against a baseline to produce the observed
symptom score `Sx`.

The production implementations (`original`, `original-extended`,
`tidyverse`) build each factor from a modified Gompertz curve
`y = maxr * (exp(-disp * exp(-rate * t)) - exp(-disp)) /
(1 - exp(-disp))` evaluated at the appropriate time index (`tod`
for BR, `tpb` for PB, cumulative `t_wk` for TV). The PB mean is
scaled by the expectancy vector `e`.

The `simple` implementation collapses this to a single linear
dose response in `br_rate` with an additive carryover adjustment
(`simulation.R:126-134`), which is adequate for pedagogy but
discards within-phase curvature.

### 2.2 Covariance structure

In the three MVN-capable implementations the 4k-by-4k covariance
matrix encodes, for k timepoints:

- AR(1) within-factor autocorrelation `rho^|t_i - t_j|` where
  `rho` is one of `c.tv`, `c.pb`, `c.br` (doc 06; root
  `generateData.R:228-242`; `original/generateData.R:188-200`;
  `ext/generateData.R:230-242`;
  `tid/functions.R:214-226`).
- Cross-factor same-timepoint correlation `c.cf1t` and
  cross-factor cross-timepoint correlation `c.cfct * rho^|dt|`
  (root `generateData.R:243-262`; `tid/functions.R:228-250`).
- Biomarker-BR differential correlation: `c.bm` on drug, decaying
  as `c.bm * exp(-lambda_cor * tsd)` off drug, only under
  Architecture B.

`lambda_cor` is resolved as `log(2) / carryover_t1half` when
`carryover_t1half > 0` and zero otherwise (root
`generateData.R:172-177`; `orig:131-135`; `ext:172-177`;
`tid/functions.R:65-73`).

Positive definiteness is checked with
`corpcor::is.positive.definite` and, on failure, silently
corrected via `corpcor::make.positive.definite(sigma, tol=1e-3)`
in all three MVN implementations. No implementation flags the
user that the returned draws come from a corrected covariance
rather than the requested one.

Sigma caching is implemented in `generateSimulatedResults` (root
and extended data.table pipelines) and in
`generate_simulated_results` (tidyverse). The cache is keyed by
the tuple (trial design, respparamset, blparamset,
modelparamset, path).

### 2.3 DGP architecture switch

| Implementation | `dgp_architecture` supported | Default |
|---|---|---|
| root (R/) | yes | mvn |
| original | no (absent; fixed MVN) | n/a |
| original-extended | yes | mvn |
| tidyverse | yes | mvn |
| simple | no (hard-coded mean moderation) | n/a |

Under `"mean_moderation"` (Architecture A), BR is additively
shifted after MVN draw:

`BR_on_drug += c.bm * (bm - bm_mean) / bm_sd * br_sd`

The shift is applied only at on-drug timepoints. Evidence:

- root `generateData.R:111-129`
- ext `generateData.R:111-129`
- tid `functions.R:419-433`
- simp `simulation.R:129-133` (multiplicative form:
  `br = br_rate * (1 + biomarker_mod * biomarker)`; algebraically
  equivalent to the additive form once `br_rate` and `br_sd` are
  reconciled, but not identical numerically.)

Under `"mvn"` (Architecture B) the interaction is encoded in the
covariance matrix and no post-hoc shift is applied. The BM-BR
correlation block in `buildSigma` is guarded on
`dgp_architecture == "mvn"` in the extended and tidyverse
implementations (ext `generateData.R:265`;
tid `functions.R:253-278`). In `original` the block is
unconditional because the implementation is fixed to B.

### 2.4 Carryover model

All production implementations parameterize carryover via a
half-life `carryover_t1half` in weeks and an exponential decay
rate `lambda_cor = ln(2) / carryover_t1half`. The DGP applies
decay to:

- BR mean accumulation: `brmeans[p] += brmeans[p-1] *
  (1/2)^(tsd[p] / carryover_t1half)` (root
  `generateData.R:201-202`).
- BM-BR correlation during off-drug periods (Architecture B
  only).

The tidyverse implementation generalizes the decay function to
three forms (`exponential`, `linear`, `weibull`) via
`carryover_decay()` (`functions.R:47-63`). The data.table
implementations expose only the exponential form.

`tsd` (time since discontinuation) is constructed in
`buildtrialdesign` by cumulatively summing interval length
`t_wk` during off-drug periods and zeroing out participants or
paths that are never on drug (root and copies
`buildtrialdesign.R:92-105`).

Boundary handling: when `carryover_t1half == 0`, all
implementations set `lambda_cor = 0`, which turns the
differential correlation into a constant `c.bm` with no decay.
No implementation validates `carryover_t1half > 0`; a negative
value would produce exponential growth in the correlation
argument (root `generateData.R:172-177`).

### 2.5 Trial designs supported

`buildtrialdesign` (root and data.table copies) and
`build_trial_design` (tidyverse) accept `timepoints`,
`timeptnames`, `expectancies`, and `ondrug` (a list of binary
path vectors) and return `trialpaths` (a list of per-path
data.tables or tibbles) with columns `timeptname`, `t_wk`, `e`,
`tod`, `tsd`, `tpb`. Designs are specified by the structure of
`ondrug`:

- Open label (OL): single path, all on-drug.
- Crossover (CO): two paths with reversed on-drug segments.
- Hybrid (OL followed by blinded randomization): mixed paths.
- OL+BDC (open label with blinded discontinuation): paths with
  intermediate on/off switches and varying expectancy.
- Parallel: two paths that are always on or always off.

The `simple` implementation hard-codes four design templates
(OL, OL+BDC, CO, N-of-1) in `create_design()`
(`simulation.R:52-79`).

### 2.6 Analysis model

In all three production implementations `lme_analysis` fits:

```
Sx ~ bm + t + Dbc + bm:Dbc        # if Db varies
Sx ~ bm + t + bm:t                # if Db does not vary
# optional additions: + De (when expectancy varies)
#                     + tsd (when simplecarryover and tsd varies)
```

with `random = ~1 | ptID` and
`correlation = nlme::corCAR1(form = ~t | ptID)`. The continuous
drug indicator `Dbc` equals 1 on drug and
`(1/2)^(carryover_scalefactor * tsd / carryover_t1half)` off
drug. The fitter retries without the correlation structure on
convergence failure (root `lme_analysis.R:159-175`;
tid `functions.R:608-635`).

The target interaction coefficient is extracted by matching
against the rowname set `{"bm:Dbc", "Dbc:bm"}` or
`{"bm:t", "t:bm"}` on `summary(fit)$tTable`. If neither is
found the function returns `NA` for the estimate, standard
error, and p-value.

The `simple` implementation replaces this with
`lm(mean_response ~ biomarker)` on phase-level summaries
(`simulation.R:235-321`). This discards within-participant
correlation, ignores AR(1) residual structure, and produces
power estimates that are upper bounds because the carryover
adjustment in `simple` uses the true (non-estimable) `br_rate`
(`simulation.R:289`).

### 2.7 Censoring / dropout

`censordata` (root and all three package-style implementations)
implements a two-pass dropout mechanism: (1) per-timepoint
dropout probabilities blending uniform and response-weighted
components, and (2) carry-forward masking so that a dropout at
timepoint `i` removes all later timepoints. The implementation
relies on `eval(parse(text=...))` at two points
(`censordata.R:50, 84`) to select delta columns by name; this
is a fragile pattern that is flagged in Section 3.2.

The `simple` implementation applies dropout by truncating at a
per-participant `dropout_week` drawn from a trial-level rate
(`simulation.R:140-210`); there is no biased dropout variant.

## 3. Code Quality and Architecture Audit

### 3.1 Design patterns

The three production implementations share a seven-module
architecture (build trial design, generate data, censor data,
fit analysis, orchestrate batch, characterize carryover, plot).
The root is a clean CRAN-style R package with `R/`, `inst/`,
`vignettes/`, `manuscripts/`, and a `tests/tinytest.R` harness.

Parallelism is implemented via `furrr::future_map` with an
auto-detection convention `n_cores = -1`, sigma caching in the
worker wrapper, and progressive RDS checkpointing when
`save_chunks = TRUE`. Reproducibility infrastructure includes a
`Makefile`, a `Dockerfile` on `rocker/tidyverse:4.5.3`, an
`renv.lock`, and the zzcollab validation targets
(`check-renv`, `check-system-deps`).

The tidyverse implementation is a faithful reimplementation that
preserves cache keys, worker semantics, and checkpoint layout
while exchanging `data.table` idioms for `dplyr`/`tibble` and
using the native pipe throughout (25 occurrences of `|>`, no
`%>%` in main code). One `magrittr` construct is retained: the
null coalescing operator `%||%` at `functions.R:319-320`.

The `simple` implementation is a single 577-line script with no
package structure, no NAMESPACE, and no tests. Its analysis
model is OLS on phase-level summaries and its carryover
adjustment references the ground-truth `br_rate`; both choices
are documented inline but not flagged as limitations in the
README.

### 3.2 Code issues

#### Finding T1: Unqualified `filter()` and `pull()` under Architecture A (critical)

Location: `implementations/tidyverse/R/functions.R:421-423`.

```
bm_mean <- baseline_param |> filter(cat == "bm") |> pull(m)
bm_sd   <- baseline_param |> filter(cat == "bm") |> pull(sd)
br_sd   <- resp_param     |> filter(cat == "br") |> pull(sd)
```

These three lines execute inside `generate_data()` only when
`dgp_architecture == "mean_moderation"`. All other references to
`dplyr::filter` and `dplyr::pull` in the tidyverse
implementation are namespace-qualified (for example lines 505,
574, 578, 605, 930, 942). If the user calls
`generate_data(..., dgp_architecture = "mean_moderation")`
without `library(dplyr)` in scope, these unqualified calls will
resolve to `stats::filter` or fail with "could not find
function 'pull'". The parity test masks this because it runs
inside a session that loads `tidyverse`.

Recommendation: qualify both calls with `dplyr::` and add a
regression test that invokes `generate_data` from a session with
only the declared `Imports` loaded.

#### Finding C1: Silent positive-definiteness correction (moderate)

Locations: root `generateData.R:284-299`;
`orig/generateData.R:242-253`; `ext/generateData.R:284-299`;
`tid/functions.R:337-356`.

When `is.positive.definite(sigma)` returns `FALSE`, all
implementations call `make.positive.definite(sigma, tol=1e-3)`
and proceed without recording that the draws come from a
corrected covariance. Doc 04 reports that 24.7 percent of
matrices in the original publication run required correction;
the current audit cannot verify the present rate without
instrumentation.

Recommendation: return a `pd_corrected` flag and aggregate it
into the per-paramset summary so that users can filter cells
whose draws depend on correction.

#### Finding C2: Hard-coded `issingular = FALSE` (moderate)

Locations: root `lme_analysis.R:178, 221`;
`orig/lme_analysis.R` same lines; ext same lines;
`tid/functions.R:654`.

The column is always `FALSE` in the returned data.table or
tibble, even when the underlying fit is singular. `nlme::lme`
exposes `fit$apVar` and `fit$modelStruct` from which singularity
can be inferred; `lme4::isSingular` is available if a compatible
fit is obtainable. As currently written, downstream consumers of
the results cannot distinguish a clean fit from a singular one.

Recommendation: detect singularity from `fit$apVar` being a
character string "Non-positive-definite approximate variance-
covariance" or from zero eigenvalues of the random-effects
variance, and populate `issingular` accordingly.

#### Finding C3: `eval(parse(text=...))` for column selection (moderate)

Locations: `censordata.R:50, 84`; `plottingfunctions.R:136`;
`lme_analysis.R:93` (partial).

These constructs compose R expressions from strings assembled
at runtime to select columns such as `D_V1`, `D_V2`. They
propagate errors late and are difficult to audit. The idiomatic
replacements are `get()`, `dat[[name]]`, or
`dplyr::all_of(name)`.

Recommendation: replace with direct indexing. No user-facing
behavior should change.

#### Finding C4: Positional coefficient extraction (low)

Locations: root `lme_analysis.R:193-217`; copies in `orig`,
`ext`; tid `functions.R:637-650`;
`carryover_analysis.R:71-93`.

Coefficient extraction relies on
`intersect(c("bm:Dbc","Dbc:bm"), rownames(summary(fit)$tTable))`
followed by `target[1]`. The construction tolerates the two
canonical orderings but would fail silently if `nlme` emitted a
different label. The `carryover_analysis` variant handles more
orderings and is the safer pattern.

#### Finding C5: Unvalidated `carryover_t1half` (low)

Locations: root `generateData.R:172-177`; copies.

A negative `carryover_t1half` produces a negative
`lambda_cor`, which turns `exp(-lambda_cor * tsd)` into an
exponentially growing correlation argument. The function does
not validate the input.

Recommendation: `stopifnot(carryover_t1half >= 0)` at the top
of `generateData`.

#### Finding C6: Stale `Imports` (low)

`lme4`, `lmerTest`, `ggpubr`, `gridExtra` appear in root
DESCRIPTION `Imports` but are not called anywhere under `R/`,
`vignettes/`, or the three package-style implementations.
These are likely residue from the pre-`nlme` analysis pipeline
and the earlier plotting scaffolding.

### 4.1 Seven DGP corrections, per-implementation status

All seven corrections enumerated in doc 03 are embodied in the

|---|---|---|---|---|
| 1. AR(1) autocorrelation (rho^|dt|) | yes | yes | yes | n/a |
| 2. Exponential BM-BR decay | yes | yes | yes | n/a |
| 3. `nlme::lme` + `corCAR1` | yes | yes | yes | no (OLS) |
| 4. Carryover scale factor removal | yes | yes | yes | n/a |
| 5. Timepoint-1 guard removal | yes | yes | yes | n/a |

### 4.2 original to original-extended

Changes are confined to three files and add Architecture A
support without altering Architecture B behaviour. Net delta
is +52 lines.

- `generateData.R` gains the `dgp_architecture` argument
  (signature), a `match.arg` enforcement (line 72), the
  Architecture A additive shift block (lines 111-129), and
  passes the argument to `buildSigma`.
- `buildSigma` (within `generateData.R`) gains a
  `dgp_architecture` argument and guards the BM-BR correlation
  block on `dgp_architecture == "mvn"` (line 265).
- `generateSimulatedResults.R` threads the argument through the
  worker wrapper (lines 74, 131, 168, 186, 196, 268, 292).

The other five files (`buildtrialdesign`, `lme_analysis`,
`utilities`, `censordata`, `carryover_analysis`) are
byte-for-byte identical.

### 4.3 original-extended to tidyverse

The tidyverse implementation is a complete reimplementation.
Parity with original-extended is enforced by
`test-parity-extended-tidyverse.R` at tolerances of `1e-10`
(sigma, data) and `1e-6` (beta, SE, p-value). The committed
`parity_baseline.rds` and the most recent `parity_report.rds`
are two bytes apart, indicating stable parity at the default
grid.

Functional differences beyond idiom translation:

- The tidyverse `carryover_decay()` exposes three forms
  (exponential, linear, Weibull). `ext` exposes only
  exponential. The `dgp_architecture == "mvn"` parity test
  exercises the exponential form only.
- Harmonization helpers `ensure_timepoint_name` and
  `ensure_timeptnames` reconcile column-name aliases
  (`timeptname` vs `timeptnames` vs `timepoint_name`) across
  the two pipelines (`functions.R:75-99`). This is necessary
  because the pipelines disagree on the canonical column name;
  downstream callers must be alias-aware.
- The tidyverse `lme_analysis` returns a tibble; `ext` returns
  a `data.table`. Both carry the same columns
  (`beta`, `betaSE`, `p`, `issingular`, `warning`).

No expected divergences are recorded in
`parity_divergences.csv` (zero data rows). This is consistent
with the observed two-byte drift between baseline and report.

### 4.4 Simplifications in `simple`

Relative to `original-extended`, `simple` omits:

- LME and `corCAR1` residual structure (uses OLS on phase-level
  means).
- Random slopes for carryover.
- Sigma caching and all covariance infrastructure.
- `furrr` parallelism (serial `replicate`).
- Architecture B.
- Biased dropout (only interval truncation).
- `carryover_analysis` half-life search and clinical threshold
  utilities.

These omissions are consistent with a pedagogical sandbox. Two
caveats qualify its claim to architectural faithfulness:

1. Power estimates under the carryover adjustment use the true
   `br_rate` (`simulation.R:289`), which is not estimable from
   data. The reported power with adjustment is therefore an
   upper bound, not a realistic estimate.
2. The "base R" framing in the README is imprecise: the script
   loads `tidyverse` at line 13 and uses `tibble()`, `map2_dfr`,
   `pmap_dbl`, and `ggplot2` throughout.

Both caveats should be added to the `simple/README.md`.

### 4.5 Unresolved divergences

- None known between `original-extended` and `tidyverse` at
  the default parity grid (baseline is stable; zero annotated
  divergences).
- The tidyverse `carryover_decay` generalizations (linear,
  Weibull) have no counterpart in the data.table
  implementations; this is a documented extension, not a
  divergence, but users combining the two pipelines in a single
  sensitivity analysis should be aware.
- The `simple` multiplicative shift `br * (1 + c.bm * bm)`
  (`simulation.R:129-133`) and the package additive shift
  `BR += c.bm * bm_z * br_sd` are algebraically distinct;
  equivalence requires aligning `br_rate` with `br_sd` and
  using centered, standardized biomarker. This is adequate for
  pedagogy but should be called out if results from `simple`
  are ever contrasted quantitatively with the package.

### 5.4 Consolidation strategy

Four live implementations create a maintenance burden
proportional to their number. The recommended long-term
arrangement is:

1. Retain `original-extended` as the canonical data.table
   production collection and promote its R files as the source
   of truth for the installed package (the root `R/` is
   effectively a copy of this collection plus plotting and
   documentation files).
2. Retain `tidyverse` as a live alternative style, with
   `test-parity-extended-tidyverse.R` as the contractual gate
   for every release. Any future divergence (for example the
   proposed linear and Weibull decay forms) should be recorded
   in `parity_divergences.csv` rather than removed.
3. Archive `original` to `analysis/archive/` after confirming
   that its Architecture B-only behaviour can be reproduced by
   `original-extended` at `dgp_architecture = "mvn"`. The
   parity test already covers this case.
4. Retain `simple` as a teaching artifact with explicit
   disclaimers (Section 5.3). Do not use its power estimates
   in any publication.

The parity infrastructure is already the right contract; the
remaining step is to reduce the four-way surface area to two
live implementations (data.table and tidyverse) plus a teaching
sandbox.
